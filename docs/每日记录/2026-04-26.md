### 2. 导航状态修复（非菜单路径处理）

#### 2.1 问题描述
从菜单页面切换到非菜单路径（如主题设置页面）时，一级菜单仍然保持选中的问题。

#### 2.2 问题根因
- `syncMenuState` 函数在找不到菜单匹配时没有清空状态
- 多个 watcher 在路由变化时没有处理非菜单路径
- FirstSidebar 中的本地 `activeKey` 没有同步清空
- 刷新页面时，`activeFirstMenu` 从 localStorage 恢复导致状态错误

#### 2.3 修改的文件

| 文件 | 修改内容 |
|------|----------|
| `app.ts` | `syncMenuState`：找不到匹配时设置 `activeFirstMenu = 'home'` 以触发 watch |
| `app.ts` | `fetchMenus`：无论菜单是否已加载都调用 `clearStateForNonMenuPath` |
| `app.ts` | `clearStateForNonMenuPath`：新增函数，检测并清空非菜单路径状态 |
| `FirstSidebar.vue` | `handleThemeSettings`：跳转前清空状态 |
| `FirstSidebar.vue` | `watch(activeFirstMenu)`：改为清空 `activeKey` 当 `newFirstMenu === 'home'` |
| `FirstSidebar.vue` | `watch(route.path)`：等待 `secondMenusMap` 加载后设置 `activeKey` |
| `SecondSidebar.vue` | `watch(secondMenusMap)`：菜单加载完成后检测路径类型，清空或恢复状态 |
| `SecondSidebar.vue` | `watch(secondMenusMap)`：非菜单路径清空所有状态，菜单路径恢复状态并强制触发 `activeFirstMenu` 变化 |

## 今日教训

### 1. 修改前应先读取规范文档
在实现 Admin 气泡框时，没有先读取 `.trae/rules.d/fe-rules.md`，导致：
- 使用了硬编码颜色而非 CSS 变量
- 使用了 `:global()` 修改 Ant Design 样式
- 手动定位而非使用 AntD 的 placement 属性

**教训**：每次进行前端 Vue/TypeScript 修改前，必须先读取 `fe-rules.md` 规范。

### 2. 全局思考问题
非菜单路径问题涉及多个文件的多个函数：
- `app.ts` 的 `syncMenuState`、`clearStateForNonMenuPath`、`fetchMenus`
- `FirstSidebar.vue` 的 `handleThemeSettings`、`watch(activeFirstMenu)`、`watch(route.path)`
- `SecondSidebar.vue` 的 `watch(secondMenusMap)`

需要从全局角度思考状态的流转，确保每个入口点都正确处理。

### 3. Watch 触发条件
当设置的新值与旧值相同时，Vue 的 watch 不会触发。可以采用：
- 先清空再设置的策略
- 或者使用 `{ deep: true }` 或 `{ immediate: true }`
