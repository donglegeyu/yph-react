## 材料申请系统 - 导航菜单选中逻辑修复

### 核心问题
左侧导航、TabBar、二级面板的选中逻辑混乱，key 不一致导致高亮失效。

### 修复内容

#### 1. key 规范化（settings-center vs setting-center）
- **问题**：API 返回 key `settings-center`（有s），前端默认值 key `setting-center`（无s）
- **修复**：`normalizeMenuKey()` 函数统一转换，三处调用：
  - `fetchNavMenusFromAPI` 构建 `firstMenus` 时
  - `secondMenusMap` 存储时
  - `selectFirstMenu` 访问 `secondMenusMap` 时

#### 2. activeFirstMenu 不规范化
- **问题**：`firstMenus.value` 中 `menu.key` 是 API 原始值（`settings-center`）
- **修复**：`activeFirstMenu` 存储原始 key，与 `firstMenus` 的 `menu.key` 直接比较

#### 3. 路由映射逻辑修正
- **问题**：`expandMenus` 存三级 key，二级菜单展开判断用二级 key，永远对不上
- **修复**：
  - `expandMenus: [level2.key]` — 存二级父级 key（用于展开判断）
  - `secondMenu: level3.key` — 存三级叶子 key（用于高亮）

#### 4. selectFirstMenu 防重复
- **问题**：TabBar 路由变化时 `selectFirstMenu` 被调用，`prevKey === key` 导致提前 return
- **修复**：`prevKey` 先保存再赋值，确保同 key 也执行；仅在不同 key 时才自动展开

## 导航菜单点击不显示导航问题

### 根因
动态添加的路由（如 `/qingwenwo`）使用 `BuildingView` 时，路由是 `default` 的直接子路由，没有经过 `MainLayout`，导致只有空白页面无导航。

### 修复内容

#### 1. NotFound 路由移入 MainLayout children
- **问题**：顶层的 `/:pathMatch(.*)*` 通配符路由在 `MainLayout` 之外，未匹配路由直接走顶层 BuildingView
- **修复**：删除顶层 `NotFound` 路由，在 `MainLayout.children` 末尾添加 `/:pathMatch(.*)*` 作为兜底

#### 2. 动态路由 BuildingView 改为异步导入
- **问题**：`router.addRoute('default', { component: BuildingView })` 中 `BuildingView` 是已解析的组件对象，与其他子路由的异步函数格式不一致
- **修复**：改为 `() => import('@/views/common/BuildingView.vue')`

#### 3. SecondSidebar 菜单点击传递 path
- **问题**：`findMenuPath(key)` 查找不到动态添加的菜单 key
- **修复**：三级菜单 `@click` 改为 `handleMenuClick(child.key, child.path)`，直接传入 path

### 相关文件
- `src/router/index.ts`：NotFound 路由移入 MainLayout children；BuildingView 改为异步导入
- `src/components/layout/SecondSidebar.vue`：handleMenuClick 支持 path 参数，三级菜单点击传递 child.path
- `src/stores/app.ts`：normalizeMenuKey 模块级化、路由映射 expandMenus/secondMenu 修正、删除重复定义

## ListPageTemplate 页面开发规范（经验总结）

### 模板文件
`src/views/common/JiuHaoHang.vue` 作为标准参考模板（导航管理 /dao-hang）

### 必须完成的 6 个 Checklist

#### 1. filterModelValue 绑定（防崩溃）
```ts
// ❌ 错误：只传 filterItems，不绑 filterModelValue
< ListPageTemplate :filter-items="filterItems" ... />

// ✅ 正确：v-model 双向绑定
const filterParams = reactive<Record<string, any>>({})
< ListPageTemplate v-model:filter-model-value="filterParams" :filter-items="filterItems" ... />
```
**原因**：FilterForm 内部访问 `modelValue[item.key]`，filterModelValue 为 undefined → 崩溃白屏

#### 2. dataSource 和 loading（显示数据）
```ts
const loading = ref(false)
const dataSource = ref<any[]>([])
< ListPageTemplate :data-source="dataSource" :loading="loading" ... />
```

#### 3. 初始数据加载（onMounted + fetch）
```ts
onMounted(() => { fetchData() })
async function fetchData() {
  loading.value = true
  try {
    const res = await fetch('/api/xxx')  // /api 前缀走 Vite 代理
    const json = await res.json()
    if (json.code === 200 && json.data) {
      dataSource.value = json.data
    }
  } finally {
    loading.value = false
  }
}
```

#### 4. 列 dataIndex 与后端返回字段严格对齐
**必须先用 curl 测试真实 API 返回的字段名**，不能靠猜：
```bash
curl -s http://localhost:8080/api/nav-menus | python3 -m json.tool
```
后端返回 `menuKey` → 前端 column 用 `dataIndex: 'menuKey'`

#### 5. 树形数据需扁平化
```ts
function flattenTree(data: any[], result: any[] = []): any[] {
  for (const item of data) {
    const { children, ...rest } = item
    result.push(rest)
    if (children?.length) flattenTree(children, result)
  }
  return result
}
// GET /api/nav-menus 返回树 → a-table 需扁平数组
dataSource.value = flattenTree(json.data)
```

#### 6. 路由注册 + pathComponentMap 映射
```ts
// router/index.ts
{ path: 'dao-hang', name: 'DaoHang', component: () => import('@/views/common/JiuHaoHang.vue') }
// pathComponentMap.ts
'/dao-hang': '@/views/common/JiuHaoHang.vue'
```

### 字段名不一致 Bug 发现与修复
- **NavMenuVO** 用 `key`，**NavMenu Entity** 用 `menuKey` → 前后端不一致
- 修复：统一 NavMenuVO 字段为 `menuKey` + service 中 `vo.setMenuKey(m.getMenuKey())`
- 教训：**DTO/VO 和 Entity 字段名必须对齐**，不能用不同的名字表示同一个数据库字段
